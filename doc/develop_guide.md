# 开发协作规范

## 1. 分支结构说明

### 1.1 核心分支
- **main**: 生产稳定分支
  - 只包含已发布的生产代码
  - 只能通过Pull Request（PR）合并
  - 受保护分支，禁止直接推送
  - 触发GitLab CI/CD部署到生产环境

- **develop**: 集成分支
  - 日常开发集成分支
  - 所有功能开发完成后合并至此
  - 触发GitLab CI/CD部署到开发/测试环境

### 1.2 临时分支
- **feature/\***: 功能分支
  - 格式：`feature/功能名称-日期` 或 `feature/issue编号-简短描述`
  - 示例：`feature/user-auth-20231015` 或 `feature/#123-add-login`
  - 从`develop`分支创建
  - 开发完成后合并回`develop`
  - **不触发自动部署**

- **hotfix/\***: 紧急修复分支
  - 格式：`hotfix/问题描述-日期`
  - 示例：`hotfix/fix-crash-20231015`
  - 从`main`分支创建
  - 修复完成后合并到`main`和`develop`
  - **不触发自动部署**

## 2. 开发流程

### 2.1 功能开发流程
```
1. 从develop创建功能分支
   git checkout develop
   git pull origin develop
   git checkout -b feature/your-feature-name

2. 在功能分支上开发
   # 进行开发工作，多次提交

3. 推送到远程仓库
   git push origin feature/your-feature-name

4. 创建合并请求（Pull Request）
   - 目标分支：develop
   - 添加描述和审查者
   - 等待CI检查通过

5. 代码审查和合并
   - 至少1人审查通过
   - 解决冲突（如有）
   - 合并到develop分支
```

### 2.2 热修复流程
```
1. 从main创建热修复分支
   git checkout main
   git pull origin main
   git checkout -b hotfix/issue-description

2. 进行修复和测试
   # 修复问题并充分测试

3. 创建两个合并请求
   PR1: hotfix → main
   PR2: hotfix → develop

4. 合并顺序
   - 先合并到main（部署到生产）
   - 再合并到develop（保持同步）
```

## 3. 提交规范

### 3.1 提交信息格式
```
类型(范围): 描述

正文（可选）

脚注（可选）
```

### 3.2 类型说明
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 重构代码
- `test`: 测试相关
- `chore`: 构建过程或辅助工具变更

### 3.3 示例
```
feat(auth): 添加用户登录功能

- 实现JWT token认证
- 添加用户登录接口
- 更新相关文档

Closes #123
```

## 4. 合并请求（PR）规范

### 4.1 PR标题格式
```
[类型] 简要描述
```
示例：`[Feature] 添加用户管理系统`

### 4.2 PR描述模板
```markdown
## 变更说明
简要描述本次变更的内容

## 相关Issue
关联的问题编号：#123

## 测试说明
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试步骤...

## 检查清单
- [ ] 代码已自检
- [ ] 文档已更新
- [ ] 无编译警告
- [ ] 遵循代码规范

## 截图（如适用）
```

## 5. CI/CD部署说明

### 5.1 自动触发规则
- **develop分支**: 
  - 每次合并后自动构建
  - 部署到开发/测试环境
  - 运行完整测试套件

- **main分支**:
  - 每次合并后自动构建
  - 部署到生产环境
  - 运行生产环境测试

### 5.2 手动部署
如需手动触发部署，可在GitLab CI/CD流水线页面操作。

## 6. C++开发注意事项

### 6.1 代码规范
- 遵循项目代码规范（如有）
- 使用`clang-format`统一格式
- 禁止提交编译警告

### 6.2 编译检查
```bash
# 本地预提交检查
mkdir build && cd build
cmake ..
make -j$(nproc)
make test  # 运行测试
```

### 6.3 依赖管理
- 更新依赖时在PR中说明
- 重大版本变更需单独评估

## 7. 冲突解决

### 7.1 基本原则
- 及时合并上游变更
- 小步提交，减少冲突
- 解决冲突后充分测试

### 7.2 解决步骤
```bash
# 1. 拉取最新代码
git fetch origin
git rebase origin/develop

# 2. 解决冲突
# 编辑冲突文件...

# 3. 继续rebase
git add .
git rebase --continue

# 4. 强制推送（功能分支）
git push origin feature/xxx --force-with-lease
```

## 8. 紧急情况处理

### 8.1 生产问题
1. 立即创建`hotfix`分支
2. 优先修复问题
3. 遵循热修复流程
4. 事后分析根本原因

### 8.2 CI/CD故障
1. 检查流水线日志
2. 回滚有问题的提交
3. 修复构建脚本
4. 重新触发部署

## 9. 最佳实践

### 9.1 分支管理
- 及时删除已合并的分支
- 保持分支生命周期短
- 避免在临时分支堆积大量变更

### 9.2 提交策略
- 原子提交（一次提交一个完整变更）
- 有意义的提交信息
- 频繁提交，定期推送

### 9.3 代码审查
- 认真对待审查意见
- 审查他人代码要仔细
- 使用GitLab的评论功能讨论
